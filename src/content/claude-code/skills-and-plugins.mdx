# 🧩 스킬과 플러그인

Claude Code는 **스킬(Skill)**과 **플러그인** 시스템으로 기능을 무한히 확장할 수 있어요. 마치 스마트폰에 앱을 설치하는 것처럼요!

## 💡 스킬이란?

스킬은 Claude Code에 **특정 능력을 부여하는 명령어 세트**예요. 슬래시 명령(`/`)으로 실행해요.

### 기본 사용법
```
> /ship          ← PR을 만들고 배포 준비
> /review        ← 코드 리뷰를 수행
> /qa            ← QA 테스트를 실행
> /investigate   ← 버그를 체계적으로 조사
```

### 스킬이 하는 일
일반 대화와 달리, 스킬은 **정해진 절차**에 따라 체계적으로 작업해요:

```
/ship 스킬 실행 시:
1. 변경사항 확인
2. 테스트 실행
3. 코드 리뷰
4. 커밋 메시지 작성
5. PR 생성
6. 결과 보고
```

## 📋 주요 스킬 소개

### 개발 워크플로우
| 스킬 | 기능 | 설명 |
|------|------|------|
| `/ship` | 배포 | PR 생성, 변경사항 정리, 테스트 확인 |
| `/review` | 코드 리뷰 | 보안, 성능, 구조 문제 분석 |
| `/investigate` | 버그 조사 | 체계적 디버깅 (원인 파악 → 수정) |
| `/qa` | QA 테스트 | 웹사이트를 자동으로 테스트하고 버그 수정 |

### 코드 품질
| 스킬 | 기능 | 설명 |
|------|------|------|
| `/health` | 건강 점검 | 코드 품질 점수와 개선 방향 제시 |
| `/simplify` | 코드 정리 | 중복 제거, 리팩토링, 효율 개선 |
| `/checkpoint` | 체크포인트 | 작업 상태를 저장하고 복원 |

### 기획과 리뷰
| 스킬 | 기능 | 설명 |
|------|------|------|
| `/office-hours` | 아이디어 검증 | YC 스타일 질문으로 아이디어를 검증 |
| `/plan-eng-review` | 기술 리뷰 | 아키텍처, 성능, 확장성 검토 |
| `/design-review` | 디자인 리뷰 | 시각적 일관성, 접근성 검토 |

## 🔌 플러그인 시스템

플러그인은 스킬보다 더 넓은 범위의 확장이에요. Claude Code의 **핵심 동작을 변경**하거나 **새로운 도구를 추가**할 수 있어요.

### Hooks
특정 이벤트가 발생할 때 자동으로 실행되는 스크립트예요:

```json
{
  "hooks": {
    "pre-commit": "npm run lint && npm test",
    "post-edit": "npx prettier --write ${file}"
  }
}
```

- **pre-commit**: 커밋 전에 린트와 테스트 자동 실행
- **post-edit**: 파일 수정 후 자동 포매팅

### 커스텀 도구
외부 서비스와 연동하는 새로운 도구를 추가할 수 있어요:
- Jira 이슈 관리
- Slack 알림 전송
- 데이터베이스 조회

## 🛠️ 나만의 스킬 만들기

### 스킬의 구조
스킬은 마크다운 파일로 정의해요:

```markdown
# 스킬 이름

## 트리거
이런 상황에서 활성화됩니다:
- "테스트를 실행해 줘"라고 요청할 때
- /test 명령을 사용할 때

## 절차
1. 프로젝트의 테스트 프레임워크를 확인합니다
2. 테스트를 실행합니다
3. 실패한 테스트가 있으면 원인을 분석합니다
4. 결과를 보고합니다
```

### 좋은 스킬의 조건
- **명확한 트리거**: 언제 실행되는지 분명해야 해요
- **단계별 절차**: 체계적인 작업 순서가 있어야 해요
- **결과 보고**: 무엇을 했는지 명확히 알려줘야 해요
- **에러 처리**: 문제가 생겼을 때의 대응이 있어야 해요

## 📦 스킬 관리

### 스킬 설치 위치
```
~/.claude/skills/          ← 글로벌 스킬 (모든 프로젝트에서 사용)
./claude/skills/           ← 프로젝트 스킬 (해당 프로젝트에서만 사용)
```

### 스킬 우선순위
프로젝트 스킬이 글로벌 스킬보다 우선돼요. 같은 이름의 스킬이 있으면 프로젝트 것이 실행돼요.

## 💡 활용 팁

1. **자주 반복하는 작업**을 스킬로 만드세요
2. **팀 규칙**을 스킬에 담으면 모든 팀원이 일관되게 작업할 수 있어요
3. 스킬을 **Git에 포함**시켜 팀과 공유하세요
4. **작은 것부터** 시작해서 점진적으로 확장하세요

## 💬 시나리오 예시

### 예시 1: 배포 워크플로우

**상황**: 기능 개발이 끝나고 PR을 만들어야 한다.

> ❌ **Before (수동 절차)**
> 린트 실행 → 테스트 실행 → 커밋 메시지 작성 → 브랜치 푸시 → GitHub에서 PR 생성. 각 단계를 직접 수행한다.

> ✅ **After (/ship 스킬)**
> `/ship`을 입력하면 변경사항 확인 → 테스트 → 코드 리뷰 → 커밋 → PR 생성까지 체계적으로 자동 수행한다.

### 예시 2: 코드 품질 점검

**상황**: 리팩토링 후 코드 품질을 확인하고 ��다.

> ❌ **Before (도구별 개별 실행)**
> ESLint, 타입 체크, 테스트, 커버리지를 각각 실행하고 결과를 직접 종합한다.

> ✅ **After (/health 스킬)**
> `/health`를 입력하면 모든 품질 도구를 실행하고 0~10점 종합 점수와 개선 방향을 한 번에 제시한다.

### 예시 3: 팀 워크플로우 표준화

**상황**: 팀원마다 코드 리뷰 관점이 달라 품질이 들쭉날쭉하다.

> ❌ **Before (개인별 리뷰)**
> 각 팀원이 자기 방식대로 리뷰한다. 보안은 확인하지만 성능은 놓치는 식의 편차가 생긴다.

> ✅ **After (커스텀 스킬)**
> 팀 리뷰 기준(보안·성능·테스트·접근성)을 스킬로 정의해 Git에 공유한다. 모든 팀원이 `/review`로 동일한 기준으로 리뷰한다.

## 📖 핵심 용어

| 용어 | 설명 | 예시 |
|------|------|------|
| **스킬** | 슬래시 명령으로 실행하는 체계적 작업 절차 | `/ship`, `/review`, `/qa` |
| **Hook** | 특정 이벤트 발생 시 자동 실행되는 스크립트 | 커밋 전 린트, 수정 후 포매팅 |
| **슬래시 명령** | `/`로 시작하는 스킬 실행 명령 | `/investigate` |
| **플러그인** | Claude Code의 핵심 동작을 변경·확장하는 시스템 | 커스텀 도구, 외부 연동 |
| **트리거** | 스킬이 활성화되는 조건 | "테스트를 실행해 줘" 요청 시 |

## 📌 핵심 정리

- 스킬은 `/명령어`로 실행하는 **체계적인 작업 절차**예요
- Hooks로 특정 이벤트에 **자동 실행**을 설정할 수 있어요
- 마크다운 파일로 **나만의 스킬**을 쉽게 만들 수 있어요
- 팀 규칙을 스킬로 만들면 **일관된 워크플로우**를 유지할 수 있어요
