# 🎛️ 하네스 엔지니어링

하네스 엔지니어링은 AI 에이전트를 실제 환경에서 **안전하고 효과적으로 운영**하기 위한 통제 시스템을 설계하는 기술이에요.

## 🐎 하네스란?

하네스(Harness)는 원래 말의 고삐와 안장을 뜻해요. AI에서의 하네스는 에이전트의 **행동을 통제하고 방향을 잡아주는 프레임워크**예요.

```
[사용자] → [하네스] → [에이전트] → [도구/환경]
              ↑                         |
              └── 모니터링 ←────────────┘
```

에이전트가 아무리 똑똑해도, 하네스 없이는 안전하게 운영할 수 없어요.

> 💡 **비유로 이해하기**: 하네스는 말의 고삐에서 이름을 따왔는데, 더 현대적인 비유로는 **자동차의 안전 시스템**이에요. 자동차(에이전트)가 아무리 성능이 좋아도, ABS(긴급 정지), 에어백(실패 복구), 차선 이탈 경고(가드레일), 블랙박스(로깅) 같은 안전장치 없이 고속도로를 달리면 위험하죠. 하네스는 AI 에이전트에게 이런 **안전 시스템을 구축**하는 기술이에요.

## 🏗️ 하네스의 핵심 구성 요소

### 1. 권한 관리 (Permission Layer)
에이전트가 할 수 있는 것과 없는 것을 정의해요:

```yaml
permissions:
  file_system:
    read: true
    write: "src/ 하위만"
    delete: false
  shell:
    allowed_commands: ["npm test", "npm run build"]
    blocked_commands: ["rm -rf", "sudo"]
  network:
    allowed_domains: ["api.github.com"]
```

### 2. 승인 게이트 (Approval Gate)
위험도에 따라 자동/수동 승인을 나눠요:

| 위험도 | 예시 | 처리 |
|--------|------|------|
| 낮음 | 파일 읽기, 검색 | 자동 승인 |
| 중간 | 파일 수정, 코드 실행 | 알림 후 자동 |
| 높음 | 파일 삭제, DB 변경 | 사람 승인 필요 |
| 치명적 | 프로덕션 배포, 결제 | 다중 승인 필요 |

### 3. 실행 샌드박스 (Sandbox)
에이전트의 실행 환경을 격리해요:
- **컨테이너 격리**: Docker 등으로 시스템 영향 차단
- **네트워크 제한**: 허용된 도메인만 접근 가능
- **리소스 제한**: CPU, 메모리, 실행 시간 제한

### 4. 모니터링과 로깅
모든 행동을 추적하고 기록해요:

```json
{
  "timestamp": "2026-04-11T10:30:00Z",
  "agent_id": "coding-agent-01",
  "action": "edit_file",
  "target": "src/auth/login.ts",
  "reasoning": "인증 로직의 타입 에러 수정",
  "status": "approved",
  "duration_ms": 1200
}
```

## 🔧 실전 하네스 패턴

### 패턴 1: 가드레일 패턴
에이전트의 출력을 필터링하는 별도의 검증 레이어를 두는 방식이에요:

```
에이전트 출력 → [가드레일 검증]
  ├→ ✅ 안전 → 실행
  ├→ ⚠️ 의심 → 사람에게 확인 요청
  └→ ❌ 위험 → 차단 + 알림
```

### 패턴 2: 체크포인트 패턴
중요한 작업 전에 상태를 저장하고, 문제 발생 시 복원해요:

```
[작업 시작]
  → 체크포인트 저장 (git stash, 스냅샷)
  → 작업 실행
  → 결과 검증
    ├→ 성공 → 체크포인트 삭제
    └→ 실패 → 체크포인트로 복원
```

### 패턴 3: 에스컬레이션 패턴
에이전트가 판단하기 어려운 상황을 상위로 넘겨요:

```
에이전트: "이 파일을 수정해야 하는데,
  테스트가 없어서 안전성을 보장할 수 없어요.
  사람이 확인해 주시겠어요?"
```

## 📊 Claude Code의 하네스

Claude Code는 하네스 엔지니어링의 좋은 실례예요:

- **권한**: `settings.json`에서 허용할 도구와 명령을 설정
- **승인**: 파일 수정, 셸 명령 실행 전 사용자 확인
- **맥락**: `CLAUDE.md`로 프로젝트 규칙을 지속적으로 전달
- **hooks**: 특정 이벤트에 자동 실행되는 스크립트 설정

```json
{
  "permissions": {
    "allow": ["Read", "Glob", "Grep"],
    "deny": ["Bash(rm -rf *)"]
  }
}
```

> 🎯 **실무 팁 — 단계별 하네스 도입**: 처음부터 완벽한 하네스를 만들 필요는 없어요. 단계별로 도입하세요:
> 1. **1단계**: 위험 명령 차단 (rm -rf, sudo 등) — 가장 먼저
> 2. **2단계**: 파일 수정 전 승인 요청 — 사고 방지
> 3. **3단계**: 실행 로깅 — 문제 추적용
> 4. **4단계**: 체크포인트 + 자동 복구 — 운영 안정성
> 5. **5단계**: 비용/시간 한도 설정 — 비용 폭발 방지

## ⚡ 하네스 설계 체크리스트

좋은 하네스를 설계하기 위한 체크리스트예요:

- [ ] 에이전트의 **최소 필요 권한**을 정의했는가?
- [ ] **위험한 작업**에 대한 승인 프로세스가 있는가?
- [ ] **실행 환경**이 격리되어 있는가?
- [ ] 모든 행동이 **로깅**되고 있는가?
- [ ] **비용 한도**가 설정되어 있는가?
- [ ] **실패 시 복구** 메커니즘이 있는가?
- [ ] **에스컬레이션** 경로가 정의되어 있는가?

## 🔮 미래 전망

하네스 엔지니어링은 AI 에이전트가 더 자율적이 될수록 더 중요해질 거예요:

- **적응형 하네스**: 에이전트의 신뢰도에 따라 자동으로 권한 조정
- **멀티 에이전트 하네스**: 여러 에이전트 간의 상호작용을 관리
- **규제 준수 하네스**: 법적 요구사항을 자동으로 반영

> 📰 **왜 하네스가 점점 더 중요해지나요?**: 2026년 현재, AI 에이전트의 자율성은 빠르게 높아졌어요. Claude Code는 파일 수정·셸 실행·배포까지 수행하고, GitHub Copilot Coding Agent는 이슈에서 PR까지 자율 처리해요. 자율성이 높아질수록 한 번의 실수가 미치는 영향도 커져요. 프로덕션 DB를 잘못 변경하거나, 민감한 API 키를 노출하는 사고가 실제로 발생하고 있어요. 하네스는 이런 사고를 **사전에 방지하는 마지노선**이에요.

## 💬 시나리오 예시

### 예시 1: 권한 관리

**상황**: 코딩 에이전트가 프로젝트 파일을 수정한다.

> ❌ **Before (무제한 접근)**
> 에이전트가 src/ 외에 .env, 배포 스크립트까지 수정할 수 있다. 실수로 환경변수가 노출된다.

> ✅ **After (최소 권한)**
> 에이전트의 쓰기 권한을 `src/` 하위로 제한하고, `.env`·`deploy/`는 읽기도 차단한다.

### 예시 2: 체크포인트 복구

**상황**: 에이전트가 대규모 리팩토링 중 실수를 했다.

> ❌ **Before (복구 불가)**
> 50개 파일을 수정한 뒤 빌드 실패. 어디서부터 잘못됐는지 찾을 수 없다.

> ✅ **After (체크포인트 패턴)**
> 작업 전 git 체크포인트를 저장한다. 실패 시 즉시 복원하고 원인을 분석한 뒤 재시도한다.

### 예시 3: 에스컬레이션

**상황**: 에이전트가 테스트 없는 코드를 발견했다.

> ❌ **Before (독단적 판단)**
> 에이전트가 "괜찮겠지"라고 판단하고 테스트 없이 수정·배포한다. 프로덕션 장애가 발생한다.

> ✅ **After (에스컬레이션)**
> "이 파일에 테스트가 없어 안전성을 보장할 수 없습니다. 테스트 추가 후 진행할까요?"라고 사람에게 판단을 넘긴다.

## 📖 핵심 용어

| 용어 | 설명 | 예시 |
|------|------|------|
| **하네스** | 에이전트의 행동을 통제·관리하는 프레임워크 | 권한·승인·모니터링 시스템 |
| **가드레일** | 에이전트 출력을 검증·필터링하는 안전 레이어 | 위험 명령 차단, 코드 검증 |
| **승인 게이트** | 위험도에 따라 자동/수동 승인을 분기하는 장치 | 파일 삭제 시 사람 승인 필요 |
| **에스컬레이션** | 에이전트가 판단 어려운 상황을 상위로 넘기는 것 | "테스트 없이 진행할까요?" |
| **체크포인트** | 작업 전 상태를 저장해 실패 시 복원하는 기법 | git stash, 스냅샷 |

## 📌 핵심 정리

- 하네스는 AI 에이전트를 **안전하게 통제**하는 프레임워크예요
- 권한 관리, 승인 게이트, 샌드박스, 모니터링이 핵심 구성 요소예요
- 가드레일, 체크포인트, 에스컬레이션 패턴을 활용하세요
- 에이전트가 강력해질수록 **하네스의 중요성**도 커져요

> 🌟 **초보자 팁**: 하네스 엔지니어링이 어렵게 느껴질 수 있지만, 이미 많이 사용하고 있을 거예요. Git을 쓰면서 "커밋 전 빌드 확인"을 하거나, 코드 리뷰 없이 머지를 못 하게 하는 것도 하네스의 일종이에요. **"실수를 방지하는 자동화된 규칙"**이 하네스의 본질이에요. 이미 알고 있는 개발 습관을 AI 에이전트에게도 적용한다고 생각하면 됩니다.
