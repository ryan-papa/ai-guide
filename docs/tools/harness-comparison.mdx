## 4대 하네스란?

Claude Code 위에서 동작하는 대표적인 **하네스(harness) 프레임워크** 4종이 있어요. 각각 다른 철학과 강점을 가지고 있어서, 목적에 맞게 선택하거나 조합해서 사용할 수 있어요.

:::tip 비유로 이해하기
Claude Code가 **범용 프로그래머**라면, 하네스는 그 프로그래머에게 붙여주는 **직무별 업무 매뉴얼**이에요.
- **SP**는 **시니어 아키텍트의 설계 원칙**
- **OMC**는 **팀장의 프로젝트 관리 도구**
- **ECC**는 **10년차 풀스택 개발자의 노하우 전집**
- **gs**는 **스타트업 CTO의 팀 운영 시스템**
:::

## 각 하네스의 핵심 역할

### SP (Superpowers) — 설계·승인 레이어

코드를 쓰기 전에 **충분히 생각하고 설계하는** 것에 집중해요. 14개 스킬로 구성되며, 브레인스토밍→계획→TDD→코드리뷰→완료까지의 워크플로우를 규율해요.

| 핵심 스킬 | 역할 |
|-----------|------|
| `brainstorming` | 아이디어→설계 변환, 요구사항 정제 |
| `writing-plans` | 구현 계획 작성, 태스크 분해 |
| `subagent-driven-development` | 태스크별 서브에이전트 파견 + 2단계 리뷰 |
| `test-driven-development` | RED-GREEN-REFACTOR 순환 |
| `requesting-code-review` | 코드리뷰어 서브에이전트 파견 |
| `verification-before-completion` | 완료 전 검증 명령 실행, 증거 확보 |

### OMC (Oh My ClaudeCode) — 오케스트레이션·실행

**여러 에이전트를 동시에 띄워 작업을 분배하고 실행**하는 데 특화되어 있어요. 19개 전문 에이전트와 9개 실행 모드를 갖추고 있어요.

| 핵심 기능 | 역할 |
|-----------|------|
| `team` | N개 에이전트 조율 (plan→PRD→exec→verify→fix) |
| `autopilot` | 단일 에이전트 자율 실행 |
| `ralph` | 완료까지 반복하는 끈질긴 모드 |
| `ultrawork` | 최대 병렬 실행 |
| `deep-interview` | 소크라테스식 요구사항 명확화 |

### ECC (Everything Claude Code) — 전문화 자산 라이브러리

**38개 에이전트, 156개 스킬, 12개 언어 생태계**를 갖춘 가장 포괄적인 하네스예요. 보안(AgentShield), 지속 학습, 하네스 감사까지 포함해요.

| 핵심 기능 | 역할 |
|-----------|------|
| 38개 전문 에이전트 | 언어별·도메인별 전문화 |
| 156개 스킬 | 실전 패턴 라이브러리 |
| AgentShield | 보안 스캐닝 (1,282 테스트) |
| 인스팅트 시스템 | 세션에서 패턴 자동 추출·학습 |
| 크로스 하네스 | Claude Code, Codex, Cursor, OpenCode, Gemini 모두 지원 |

### gs (gstack) — 스타트업 팀 시뮬레이션

Y Combinator CEO Garry Tan이 만든 하네스로, **Claude Code를 가상 엔지니어링 팀**으로 변환해요. CEO, 디자이너, 엔지니어, QA 역할을 시뮬레이션해요.

| 핵심 기능 | 역할 |
|-----------|------|
| 23개 슬래시 명령 | 역할별 전문가 (CEO, Designer, Engineer, QA) |
| 8개 파워 도구 | `/office-hours`, `/plan-ceo-review`, `/ship` 등 |
| 스프린트 구조 | `/office-hours`→`/plan-ceo-review`→`/review`→`/qa`→`/ship` |
| 프로덕트 철학 | "Build what matters" — 스타트업 속도로 품질 유지 |

## 워크플로우 단계별 비교

각 하네스가 개발 단계별로 어떤 명령을 제공하는지 비교한 표예요.

| 단계 | SP | OMC | ECC | gs |
|------|-----|-----|-----|-----|
| **생각 (THINK)** | `brainstorming` | `deep-interview`, `ultrathink` | — | `/office-hours` |
| **계획 (PLAN)** | `writing-plans` | `omc-plan`, `ralplan` | — | `/plan-ceo-review` |
| **구현 (BUILD)** | `executing-plans`, `subagent-driven-development` | `autopilot`, `team`, `ralph` | 38개 에이전트 자동 배정 | `/build`, `/parallel` |
| **리뷰 (REVIEW)** | `requesting-code-review` | `/ask codex`, `/ccg` | `/security-scan` | `/review`, `/design-review` |
| **테스트 (TEST)** | `test-driven-development` | `ultraqa` | AgentShield | `/qa` |
| **배포 (SHIP)** | `finishing-a-development-branch` | — | — | `/ship`, `/changelog` |

## 어떤 하네스를 선택할까?

| 상황 | 추천 | 이유 |
|------|------|------|
| 신중한 설계가 중요한 프로젝트 | **SP** | 브레인스토밍→TDD→코드리뷰 워크플로우 |
| 여러 작업을 동시에 빠르게 | **OMC** | Team 모드, 병렬 실행, Codex/Gemini 연동 |
| 다양한 언어·프레임워크 사용 | **ECC** | 12개 언어 생태계, 156개 전문 스킬 |
| 스타트업처럼 빠르게 제품 출시 | **gs** | CEO/디자이너/QA 역할 시뮬레이션 |
| **모두 조합** | SP + OMC + ECC | SP가 설계, OMC가 실행, ECC가 전문 자산 제공 |

:::info 조합 사용
4종 하네스는 레이어가 다르기 때문에 **함께 사용할 수 있어요**. 대표적인 조합:
- **SP(설계) + OMC(실행)**: SP가 계획을 세우고, OMC가 멀티 에이전트로 실행
- **OMC(오케스트레이션) + ECC(전문 자산)**: OMC가 조율하고, ECC의 언어별 스킬로 품질 향상
- **gs(프로덕트) + ECC(기술)**: gs의 스프린트 구조 + ECC의 보안·품질 체크
:::

## 핵심 용어

- **하네스(Harness)**: AI 에이전트의 행동을 규율하는 규칙·스킬·훅의 집합이에요. "마구(馬具)"에서 유래해요.
- **스킬(Skill)**: 재사용 가능한 작업 패턴을 문서화한 것. 트리거 조건이 맞으면 자동으로 로드돼요.
- **오케스트레이션(Orchestration)**: 여러 에이전트의 작업을 조율하는 것이에요.
- **서브에이전트(Subagent)**: 메인 에이전트가 특정 작업을 위임하는 독립 에이전트예요.
