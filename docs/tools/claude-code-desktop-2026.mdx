## 무엇이 바뀌었나요?

2026년 4월, Claude Code 데스크톱 앱이 대규모로 리디자인되었어요. 핵심은 **병렬 에이전트 작업**을 위한 전면 개편이에요. 하나의 작업만 기다리는 게 아니라, 여러 작업을 동시에 띄워놓고 오케스트레이터처럼 관리하는 워크플로우에 맞춰 설계되었어요.

:::tip 비유로 이해하기
기존 Claude Code가 **단일 모니터에서 하나의 터미널**을 쓰는 느낌이었다면, 새 데스크톱 앱은 **여러 모니터에 에디터, 터미널, 브라우저를 띄워놓고 작업**하는 느낌이에요. 각 에이전트 세션이 독립적으로 돌아가고, 결과가 오면 확인하고 다음 지시를 내리는 거예요.
:::

| 항목 | 내용 |
|------|------|
| **발표일** | 2026년 4월 14일 |
| **대상** | Pro, Max, Team, Enterprise 플랜 + API 사용자 |
| **다운로드** | [claude.com/download](https://claude.com/download) |

## 신규 기능 한눈에 보기

| 기능 | 설명 |
|------|------|
| **사이드바 세션 관리** | 모든 활성/최근 세션을 한 곳에서 관리. 프로젝트별 그룹핑, 상태별 필터링 |
| **병렬 세션 실행** | 여러 레포에서 동시에 작업 시작, 결과가 오면 이동하며 리뷰 |
| **드래그앤드롭 레이아웃** | 터미널, 프리뷰, 디프 뷰어, 채팅을 자유롭게 배치 |
| **내장 터미널** | 앱 안에서 직접 테스트/빌드 실행 |
| **인앱 파일 에디터** | 파일 열기, 수정, 저장까지 앱 안에서 완결 |
| **빠른 디프 뷰어** | 대규모 변경사항도 빠르게 리뷰 가능하도록 재구축 |
| **확장된 프리뷰** | HTML, PDF 파일을 앱 내에서 열기, 로컬 서버 프리뷰 |
| **사이드 채팅** | ⌘+; 로 작업 중 별도 질문 분기 (메인 스레드에 영향 없음) |
| **플러그인 호환** | CLI 플러그인이 데스크톱에서도 동일하게 작동 |
| **SSH 지원 확장** | Mac에서도 SSH로 원격 머신 세션 실행 |

## 세션 관리와 1M 컨텍스트

같은 시기에 발표된 세션 관리 가이드도 중요해요. Claude Code의 **컨텍스트 윈도우가 100만 토큰**으로 확장되면서, 세션을 어떻게 관리하느냐가 결과 품질에 큰 영향을 미치게 되었어요.

### 컨텍스트 윈도우란?

모델이 한 번에 "볼 수 있는" 전체 정보예요. 시스템 프롬프트, 대화 내역, 도구 호출 결과, 읽은 파일 등이 모두 포함돼요.

### 컨텍스트 부패 (Context Rot)

컨텍스트가 커질수록 모델 성능이 떨어지는 현상이에요. 주의(attention)가 더 많은 토큰에 분산되면서, 오래된 무관한 내용이 현재 작업을 방해하게 돼요.

:::tip 비유로 이해하기
**컨텍스트 부패**는 책상 위에 서류가 쌓이는 것과 같아요. 처음에는 필요한 문서를 바로 찾지만, 100장, 200장 쌓이면 정작 필요한 한 장을 찾기 어려워져요. 이때 **정리(compaction)**가 필요한 거예요.
:::

### 매 턴마다의 선택지

Claude가 작업을 마칠 때마다 여러 선택지가 있어요.

| 명령 | 동작 | 사용 시점 |
|------|------|-----------|
| **계속 대화** | 같은 세션에서 추가 메시지 | 관련 후속 작업 |
| `/rewind` (esc esc) | 이전 메시지로 되돌아가 다시 시도 | 결과가 마음에 안 들 때 |
| `/clear` | 새 세션 시작 | 주제가 완전히 바뀔 때 |
| `/compact` | 현재 대화를 요약 후 압축 | 컨텍스트가 너무 클 때 |
| 서브에이전트 | 별도 컨텍스트에서 독립 작업 | 탐색적 작업, 메인 오염 방지 |

### 세션 관리 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **집중된 세션** | 하나의 세션에 하나의 명확한 목표 |
| **적시 압축** | 컨텍스트가 무거워지면 `/compact`로 정리 |
| **서브에이전트 활용** | 탐색적 작업은 서브에이전트에 위임하여 메인 컨텍스트 보호 |
| **새 세션 두려워 말기** | 주제가 바뀌면 과감히 `/clear` |

## 루틴 (Routines)

같은 날 발표된 **루틴** 기능도 큰 변화예요. 루틴은 한번 설정하면 **자동으로 반복 실행**되는 Claude Code 자동화예요.

### 루틴의 세 가지 유형

| 유형 | 트리거 | 예시 |
|------|--------|------|
| **스케줄 루틴** | 시간 기반 (매시간, 매일, 매주) | 매일 새벽 2시: Linear에서 버그 꺼내서 수정하고 드래프트 PR |
| **API 루틴** | HTTP 요청 | 알림 수신 → 담당 서비스 찾기 → #oncall 채널에 요약 |
| **웹훅 루틴** | GitHub 이벤트 | PR 생성 시 자동 리뷰, CI 실패 시 자동 대응 |

### 실제 활용 사례

| 패턴 | 설명 |
|------|------|
| **백로그 관리** | 매일 밤 새 이슈 분류, 라벨링, 우선순위 지정 |
| **자동 PR 리뷰** | PR마다 세션 생성, 코멘트·CI 실패까지 추적 |
| **코드 표준 감시** | 매주 코드베이스 스캔, 위반 사항 보고 |
| **인시던트 대응** | 알림 → 원인 분석 → 트리아지 요약 자동 포스팅 |

:::info 루틴의 핵심 장점
루틴은 **Claude Code의 웹 인프라에서 실행**돼요. 노트북을 열어둘 필요가 없고, 레포와 커넥터에 접근 권한이 있어서 별도 MCP 서버나 cron job 설정이 불필요해요.
:::

```bash
# CLI에서 스케줄 설정 (예: 매일 밤 이슈 정리)
/schedule "Linear에서 상위 버그 하나를 가져와 수정하고 드래프트 PR 열어줘"
```

## 에이전트 도구 설계 철학

같은 시기에 공개된 "Seeing like an agent" 블로그에서는 Claude Code 팀이 **도구를 어떻게 설계하는지** 철학을 공유했어요. 에이전트를 활용하거나 직접 만들 때 참고할 만한 원칙이에요.

### 핵심 원칙

| 원칙 | 설명 | 예시 |
|------|------|------|
| **모델의 눈으로 보기** | 도구를 설계할 때 모델의 능력에 맞춰야 함 | 수학 문제에 계산기 vs 종이 — 모델 능력에 따라 최적 도구가 다름 |
| **마찰 줄이기** | 사용자-모델 간 소통 마찰을 최소화 | 질문에 객관식 옵션을 추가해 응답 시간 단축 |
| **점진적 노출** | 기본은 단순하게, 필요하면 고급 옵션 노출 | 검색 도구: 기본 쿼리만 필수, 정규식은 선택 |
| **의도 전달** | 도구 이름과 설명이 용도를 명확히 전달 | `TodoWrite` → `Task`로 변경하여 더 적극적 활용 유도 |

### 실전 교훈

Claude Code 팀이 도구를 만들면서 겪은 실제 시행착오예요.

**AskUserQuestion 도구 — 3번의 시도:**
1. ExitPlanTool에 질문 배열 추가 → 계획과 질문이 섞여 혼란
2. 별도 AskUserQuestion 도구 → 동작하지만 응답이 느림
3. 객관식 + 자유 입력 옵션 추가 → 소통 마찰 대폭 감소 ✅

**TodoWrite → Task 변경:**
- `TodoWrite`라는 이름일 때 모델이 수동적으로 사용 (할 일 기록용으로만)
- `Task`로 변경하자 작업 분해·위임에 적극 활용 → 코딩 벤치마크 점수 향상

## 핵심 용어

- **컨텍스트 윈도우(Context Window)**: 모델이 한 번에 처리할 수 있는 최대 토큰 수. Claude Code는 100만 토큰을 지원해요.
- **컨텍스트 부패(Context Rot)**: 컨텍스트가 커지면서 모델 성능이 저하되는 현상이에요.
- **컴팩션(Compaction)**: 긴 대화를 요약해서 새 컨텍스트로 이어가는 기능이에요. 자동 또는 `/compact`로 수동 실행해요.
- **루틴(Routine)**: 프롬프트, 레포, 커넥터를 설정해두면 자동으로 반복 실행되는 Claude Code 자동화예요.
- **사이드 채팅(Side Chat)**: 메인 작업 스레드를 오염시키지 않고 별도 질문을 할 수 있는 분기 대화예요.
- **점진적 노출(Progressive Disclosure)**: 도구의 기본 인터페이스는 단순하게 유지하고, 고급 옵션은 필요할 때만 노출하는 설계 원칙이에요.
