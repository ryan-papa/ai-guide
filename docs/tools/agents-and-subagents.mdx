# 🤖 에이전트와 서브에이전트

Claude Code는 단순한 챗봇이 아닌 **에이전트**로 동작해요. 그리고 복잡한 작업은 **서브에이전트**에게 위임할 수 있어요.

:::tip 쉬운 비유
에이전트 시스템은 **건설 현장의 감독관과 작업팀**과 비슷해요. 감독관(메인 에이전트)이 전체 설계도를 보면서 "A팀은 배관, B팀은 전기, C팀은 도장"처럼 작업을 분배하고, 각 팀(서브에이전트)은 자기 영역에 집중해요. 모든 작업이 끝나면 감독관이 결과를 종합해서 보고하죠.
:::

## 🧠 메인 에이전트의 역할

Claude Code의 메인 에이전트는 다음을 수행해요:

1. **사용자 의도 파악**: 요청을 이해하고 작업 계획 수립
2. **도구 선택**: 적절한 도구를 골라서 실행
3. **결과 분석**: 실행 결과를 확인하고 다음 행동 결정
4. **작업 위임**: 독립적인 작업을 서브에이전트에게 넘김

```
사용자: "인증 시스템을 JWT에서 세션 기반으로 변경해 줘"

메인 에이전트의 사고:
1. 현재 인증 구조를 파악해야 함 → 코드 분석
2. 영향 받는 파일들을 식별 → 검색
3. 변경 계획을 수립 → 설계
4. 코드를 수정 → 구현
5. 테스트를 실행 → 검증
```

## 🔀 서브에이전트란?

서브에이전트는 메인 에이전트가 생성하는 **독립적인 작업 단위**예요. 별도의 컨텍스트에서 특정 작업을 수행해요.

### 서브에이전트의 특징
- **독립적 컨텍스트**: 메인 에이전트와 별도의 대화 공간
- **제한된 범위**: 맡은 작업에만 집중
- **결과 반환**: 완료 후 결과를 메인 에이전트에게 보고
- **병렬 실행**: 여러 서브에이전트가 동시에 작업 가능

### 언제 서브에이전트를 쓰나요?

```
✅ 서브에이전트가 적합한 경우:
- 여러 파일을 독립적으로 분석할 때
- 코드베이스 전체를 검색할 때
- 병렬로 처리 가능한 작업들

❌ 서브에이전트가 부적합한 경우:
- 사용자와 대화가 필요한 작업
- 이전 결과에 의존하는 순차 작업
- 위험도가 높은 작업 (파일 삭제, 배포 등)
```

## 🏗️ 위임 패턴

### 패턴 1: 조사 위임
여러 영역을 동시에 조사해요:

```
메인: "이 프로젝트의 보안 취약점을 찾아줘"

서브에이전트 A: 인증/인가 코드 분석
서브에이전트 B: API 엔드포인트 점검
서브에이전트 C: 의존성 취약점 확인

→ 세 결과를 종합해서 보고
```

### 패턴 2: 변환 위임
동일한 작업을 여러 대상에 적용해요:

```
메인: "모든 API 라우트에 에러 핸들링을 추가해 줘"

서브에이전트 A: routes/auth.ts 수정
서브에이전트 B: routes/users.ts 수정
서브에이전트 C: routes/products.ts 수정

→ 각 결과를 검증하고 통합
```

### 패턴 3: 검증 위임
구현과 검증을 분리해요:

```
메인: 코드를 수정

서브에이전트(검증): "수정된 코드에 문제가 있는지 확인해"
→ 독립적인 시각으로 코드를 검토
```

## 📊 에이전트 계층 구조

```
[사용자]
    ↓
[메인 에이전트] ─── 사용자와 직접 대화
    ├── [서브에이전트 1] ── 파일 분석
    ├── [서브에이전트 2] ── 코드 검색  
    └── [서브에이전트 3] ── 테스트 실행
```

### 컨텍스트 관리
- **메인 에이전트**: 전체 대화 맥락, 사용자 의도, 프로젝트 지식을 보유
- **서브에이전트**: 메인이 전달한 작업 설명과 필요한 정보만 보유

이런 분리 덕분에:
- 메인의 컨텍스트 창이 낭비되지 않아요
- 서브에이전트는 작업에 집중할 수 있어요
- 여러 작업을 병렬로 빠르게 처리해요

## ⚡ 실전 활용 예시

### 대규모 리팩토링
```
> src 폴더의 모든 JavaScript 파일을 TypeScript로 변환해 줘

메인 에이전트:
1. 변환 대상 파일 목록 수집 (15개 파일)
2. 파일간 의존성 분석
3. 서브에이전트에게 독립 파일들 병렬 변환 위임
4. 결과 검증 및 타입 에러 해결
5. 최종 빌드 테스트
```

### 코드 리뷰
```
> 이번 PR의 변경사항을 리뷰해 줘

메인 에이전트:
1. PR diff 확인
2. 서브에이전트 A: 보안 관점 리뷰
3. 서브에이전트 B: 성능 관점 리뷰
4. 서브에이전트 C: 테스트 커버리지 확인
5. 종합 리뷰 보고서 작성
```

## 🛡️ 안전 고려사항

서브에이전트 사용 시 주의할 점:

1. **읽기 전용 권장**: 서브에이전트에게는 가능한 읽기 권한만 부여
2. **결과 검증**: 서브에이전트의 결과를 메인이 반드시 확인
3. **비용 인식**: 여러 서브에이전트 = 여러 배의 토큰 소비
4. **범위 제한**: 서브에이전트의 작업 범위를 명확히 정의

### 토큰 비용 이해하기

서브에이전트는 각각 독립된 컨텍스트를 가지기 때문에 토큰을 별도로 소비해요:

```
예시: 보안 점검 작업

순차 처리 (서브에이전트 없이):
  메인 에이전트 1개 × 약 50K 토큰 = 50K 토큰
  ⏱️ 소요 시간: 3분

병렬 처리 (서브에이전트 3개):
  메인 에이전트: 약 20K 토큰
  서브에이전트 3개 × 약 20K 토큰 = 60K 토큰
  합계: 약 80K 토큰 (1.6배)
  ⏱️ 소요 시간: 1분 (3배 빠름)

→ 토큰은 더 쓰지만, 시간은 크게 줄어요.
   속도가 중요한 상황에서 유리해요.
```

:::info 서브에이전트는 언제 자동으로 사용되나요?
Claude Code는 복잡한 작업을 감지하면 자동으로 서브에이전트를 생성해요. 사용자가 직접 "서브에이전트를 써"라고 지시할 필요는 없어요. 다만, "각 파일을 병렬로 분석해 줘"처럼 힌트를 주면 더 적극적으로 활용해요.
:::

## 💬 시나리오 예시

### 예시 1: 보안 점검

**상황**: PR 전에 프로젝트 전체 보안 점검을 해야 한다.

> ❌ **Before (순차 처리)**
> 인증 코드 점검 → API 엔드포인트 점검 → 의존성 취약점 점검을 순서대로 한다. 시간이 3배 걸린다.

> ✅ **After (서브에이전트 병렬 위임)**
> 메인 에이전트가 3개 서브에이전트에게 각 영역을 동시에 위임한다. 결과를 종합해 하나의 보안 보고서를 생성한다.

### 예시 2: 검증 분리

**상황**: 복잡한 코드를 수정한 뒤 독립적 검증이 필요하다.

> ❌ **Before (자기 검증)**
> 수정한 에이전트가 자기 코드를 직접 검증한다. 자신의 실수를 발견하기 어렵다.

> ✅ **After (검증 서브에이전트)**
> 메인이 코드를 수정한 뒤, 별도 서브에이전트가 독립적 시각으로 리뷰한다. 편향 없는 품질 검증이 가능하다.

## 📖 핵심 용어

| 용어 | 설명 | 예시 |
|------|------|------|
| **메인 에이전트** | 사용자와 대화하며 전체 작업을 관리하는 에이전트 | Claude Code 본체 |
| **서브에이전트** | 메인이 생성한 독립 작업 단위 | 파일 분석, 코드 검색 담당 |
| **위임(Delegation)** | 메인이 서브에이전트에게 작업을 넘기는 것 | 보안 점검 3개 영역 병렬 위임 |
| **병렬 실행** | 여러 서브에이전트가 동시에 작업하는 것 | 3개 파일 동시 리뷰 |
| **컨텍스트 분리** | 서브에이전트가 별도 대화 공간에서 작업하는 것 | 메인 컨텍스트 낭비 방지 |
| **토큰** | AI 모델이 텍스트를 처리하는 최소 단위 (≈ 한국어 1글자) | 서브에이전트 3개 = 토큰 3배 소비 |
| **컨텍스트 윈도우** | AI가 한 번에 기억할 수 있는 텍스트 크기 제한 | 서브에이전트로 분산하면 효율적 |

## 📌 핵심 정리

- 메인 에이전트가 **전체 작업을 관리**하고 서브에이전트에게 **위임**해요
- 서브에이전트는 **독립적 컨텍스트**에서 특정 작업을 수행해요
- 조사, 변환, 검증 등 다양한 **위임 패턴**이 있어요
- **병렬 처리**로 복잡한 작업을 빠르게 완수할 수 있어요
